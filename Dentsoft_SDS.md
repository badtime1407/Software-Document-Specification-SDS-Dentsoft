## 1. บทนำ (Introduction)

### 1.1 วัตถุประสงค์ของเอกสาร (Purpose)
เอกสารฉบับนี้จัดทำขึ้นเพื่อระบุข้อกำหนดความต้องการทั้งหมดของซอฟต์แวร์ **ระบบจัดการคลินิกทันตกรรม (Dental Clinic Management System)** โดยจะอธิบายถึงฟังก์ชันการทำงาน, คุณลักษณะของผู้ใช้, และข้อจำกัดต่างๆ ของระบบ เพื่อใช้เป็นแนวทางในการออกแบบ, พัฒนา, และทดสอบระบบต่อไป

### 1.2 ขอบเขตของโครงงาน (Project Scope)
ระบบจัดการคลินิกทันตกรรมนี้เป็นเว็บแอปพลิเคชันที่พัฒนาขึ้นเพื่อเพิ่มประสิทธิภาพในการบริหารจัดการข้อมูลภายในคลินิกทันตกรรม ประกอบด้วย 3 ส่วนหลักสำหรับผู้ใช้งาน 3 กลุ่ม คือ **คนไข้ (Patient)**, **ทันตแพทย์ (Dentist)**, และ **ผู้ดูแลระบบ (Admin)**

ระบบจะครอบคลุมการทำงานตั้งแต่การลงทะเบียนของคนไข้, การจองและยกเลิกนัดหมาย, การดูประวัติการรักษา ไปจนถึงการจัดการข้อมูลคนไข้และตารางงานของทันตแพทย์โดยผู้ดูแลระบบ และการบันทึกการรักษาโดยทันตแพทย์ นอกจากนี้ยังมีระบบ AI Chatbot เพื่อตอบคำถามเบื้องต้นและให้ข้อมูลแก่คนไข้

### 1.3 ภาพรวมของเอกสาร (Document Overview)
- **บทที่ 1** กล่าวถึงภาพรวมของเอกสารและขอบเขตของโครงงาน
- **บทที่ 2** อธิบายภาพรวมของผลิตภัณฑ์, กลุ่มผู้ใช้งาน, สภาพแวดล้อมการทำงาน และข้อจำกัดต่างๆ
- **บทที่ 3** ลงรายละเอียดข้อกำหนดเฉพาะของระบบ ทั้งในเชิงฟังก์ชัน, ส่วนต่อประสาน, และข้อกำหนดอื่นๆ ที่ไม่ใช่เชิงฟังก์ชัน

## 2. คำอธิบายโดยรวม (Overall Description)

### 2.1 มุมมองของผลิตภัณฑ์ (Product Perspective)
ระบบนี้เป็นเว็บแอปพลิเคชันแบบ Standalone ที่ทำงานผ่านเว็บเบราว์เซอร์บนอุปกรณ์คอมพิวเตอร์และอุปกรณ์เคลื่อนที่ (Responsive Design) โดยมีการเชื่อมต่อกับระบบฐานข้อมูลเพื่อจัดเก็บข้อมูลทั้งหมด

### 2.2 กลุ่มผู้ใช้งานและคุณลักษณะ (User Classes and Characteristics)

| กลุ่มผู้ใช้ | คำอธิบาย |
| :--- | :--- |
| **คนไข้ (Patient)** | บุคคลทั่วไปที่ต้องการเข้าถึงบริการของคลินิก เช่น จองนัดหมาย, ดูประวัติการรักษาของตนเอง, และสอบถามข้อมูล |
| **ทันตแพทย์ (Dentist)** | บุคลากรทางการแพทย์ผู้ให้บริการตรวจและรักษา มีหน้าที่ในการดูตารางนัดหมายของตนเองและบันทึกข้อมูลการรักษา |
| **ผู้ดูแลระบบ (Admin)** | เจ้าหน้าที่คลินิกผู้มีสิทธิ์สูงสุดในการจัดการข้อมูลในระบบ เช่น จัดการบัญชีผู้ใช้, จัดการตารางงานแพทย์, และดูรายงานสรุป |

### 2.3 สภาพแวดล้อมการทำงาน (Operating Environment)
- **Front-end:** Next.js, TypeScript, Tailwind CSS
- **Back-end:** Next.js (API Routes)
- **Database:** PostgreSQL
- **ORM:** Prisma ORM
- **Authentication:** NextAuth.js
- **Deployment:** Vercel, Supabase
- **Development Tools:** Visual Studio Code, GitHub, Postman
- **Design Tools:** Figma, Canva

### 2.4 ข้อจำกัดในการออกแบบและการพัฒนา (Design and Implementation Constraints)
- ระบบต้องพัฒนาด้วยเทคโนโลยีที่ระบุไว้ในข้อ 2.3
- ระบบต้องรองรับการใช้งานบนเว็บเบราว์เซอร์สมัยใหม่ (เช่น Google Chrome, Firefox, Safari)
- การออกแบบต้องเป็นแบบ Responsive เพื่อรองรับการแสดงผลบนขนาดหน้าจอที่แตกต่างกัน
- การพิสูจน์ตัวตนต้องทำผ่าน NextAuth.js

## 3. ข้อกำหนดเฉพาะ (Specific Requirements)

### 3.1 ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)

#### 3.1.1 ส่วนของคนไข้ (Patient)
| ID | Requirement Description |
| :--- | :--- |
| **FR-PAT-01** | ระบบต้องอนุญาตให้ผู้ใช้ใหม่สามารถสมัครสมาชิกได้ |
| **FR-PAT-02** | ระบบต้องอนุญาตให้คนไข้ที่ลงทะเบียนแล้วสามารถเข้าสู่ระบบได้ |
| **FR-PAT-03** | ระบบต้องอนุญาตให้คนไข้สามารถแก้ไขข้อมูลส่วนตัวของตนเองได้ |
| **FR-PAT-04** | ระบบต้องแสดงรายการบริการทางทันตกรรมทั้งหมดของคลินิกได้ |
| **FR-PAT-05** | ระบบต้องอนุญาตให้คนไข้สามารถจองนัดหมายโดยการเลือกบริการ, วัน, และเวลาได้ |
| **FR-PAT-06** | ระบบต้องอนุญาตให้คนไข้สามารถยกเลิกนัดหมายของตนเองได้ |
| **FR-PAT-07** | ระบบต้องแสดงข้อมูลสรุปประวัติการรักษาที่ผ่านมาของคนไข้ได้ |
| **FR-PAT-08** | ระบบต้องมีการส่งการแจ้งเตือน (Notification) ก่อนถึงวันนัดหมาย |
| **FR-PAT-09** | ระบบต้องมี AI Chatbot เพื่อให้คนไข้สอบถามข้อมูลเบื้องต้นได้ |

#### 3.1.2 ส่วนของทันตแพทย์ (Dentist)
| ID | Requirement Description |
| :--- | :--- |
| **FR-DEN-01** | ระบบต้องอนุญาตให้ทันตแพทย์สามารถเข้าสู่ระบบได้ |
| **FR-DEN-02** | ระบบต้องแสดงรายการนัดหมายทั้งหมดของทันตแพทย์คนนั้นๆ ได้ |
| **FR-DEN-03** | ระบบต้องอนุญาตให้ทันตแพทย์สามารถค้นหานัดหมายตามวันและเวลาได้ |
| **FR-DEN-04** | ระบบต้องแสดงข้อมูลของคนไข้ที่นัดหมายเข้ามาเพื่อรับบริการได้ |
| **FR-DEN-05** | ระบบต้องอนุญาตให้ทันตแพทย์สามารถบันทึกข้อมูลการรักษาได้ (เช่น ซี่ฟันที่รักษา, ปัญหา, แนวทางการรักษา) |
| **FR-DEN-06** | ระบบต้องอนุญาตให้ทันตแพทย์สามารถดูประวัติการรักษาทั้งหมดของคนไข้ได้ |

#### 3.1.3 ส่วนของผู้ดูแลระบบ (Admin)
| ID | Requirement Description |
| :--- | :--- |
| **FR-ADM-01** | ระบบต้องอนุญาตให้ผู้ดูแลระบบสามารถเข้าสู่ระบบได้ |
| **FR-ADM-02** | ระบบต้องอนุญาตให้ผู้ดูแลระบบสามารถจัดการบัญชีทันตแพทย์ได้ (เพิ่ม, แก้ไข, ลบ) |
| **FR-ADM-03** | ระบบต้องแสดงรายชื่อคนไข้ทั้งหมดในระบบได้ |
| **FR-ADM-04** | ระบบต้องอนุญาตให้ผู้ดูแลระบบจัดการตารางการทำงานของทันตแพทย์ได้ |
| **FR-ADM-05** | ระบบต้องอนุญาตให้ผู้ดูแลระบบดูและจัดการนัดหมายทั้งหมดในระบบได้ |
| **FR-ADM-06** | ระบบต้องอนุญาตให้ผู้ดูแลระบบแก้ไขสถานะของการนัดหมายได้ (เช่น ยืนยัน, เสร็จสิ้น, ยกเลิก) |
| **FR-ADM-07** | ระบบต้องอนุญาตให้ผู้ดูแลระบบสามารถเปลี่ยนวัน, เวลา, หรือทันตแพทย์สำหรับนัดหมายที่มีอยู่ได้ |
| **FR-ADM-08** | ระบบต้องอนุญาตให้ผู้ดูแลระบบสามารถสร้างนัดหมายต่อเนื่องจากการรักษาก่อนหน้าได้ |
| **FR-ADM-09** | ระบบต้องสามารถสร้างรายงานและสถิติการใช้งานระบบได้ ประกอบด้วย: <br>   - จำนวนคนไข้รายเดือน <br>   - บริการที่ได้รับความนิยมสูงสุด <br>   - รายได้รายวัน, รายสัปดาห์, และรายเดือน |

### 3.2 ข้อกำหนดด้านส่วนต่อประสานภายนอก (External Interface Requirements)

#### 3.2.1 ส่วนต่อประสานกับผู้ใช้ (User Interfaces)
- **UI-01:** ระบบจะต้องมี User Interface ที่ออกแบบมาให้ใช้งานง่าย (User-Friendly) และเป็นไปตามที่ออกแบบไว้ใน Figma
- **UI-02:** การแสดงผลจะต้องเป็นแบบ Responsive กล่าวคือ สามารถปรับเปลี่ยนการแสดงผลให้เหมาะสมกับอุปกรณ์ที่แตกต่างกันได้ เช่น Desktop, Tablet, และ Mobile Phone

#### 3.2.2 ส่วนต่อประสานกับซอฟต์แวร์ (Software Interfaces)
- **SI-01:** ระบบจะใช้ Prisma ORM ในการเชื่อมต่อและจัดการข้อมูลกับฐานข้อมูล PostgreSQL
- **SI-02:** ระบบจะใช้ NextAuth.js ในการจัดการกระบวนการยืนยันตัวตน (Authentication) และการจัดการ Session ของผู้ใช้
- **SI-03:** (ถ้ามี) ระบบอาจมีการเชื่อมต่อกับ API ภายนอก เช่น บริการส่ง SMS/Email สำหรับการแจ้งเตือน

### 3.3 ข้อกำหนดที่ไม่ใช่เชิงฟังก์ชัน (Non-Functional Requirements)

| ID | Type | Requirement Description |
| :--- | :--- | :--- |
| **NFR-01** | **Performance** | ระบบควรตอบสนองต่อการใช้งานได้อย่างรวดเร็ว โดยหน้าเว็บเพจควรโหลดเสร็จสิ้นภายใน 3 วินาที |
| **NFR-02** | **Security** | ข้อมูลรหัสผ่านของผู้ใช้จะต้องถูกเข้ารหัส (Hashing) ก่อนจัดเก็บลงฐานข้อมูล |
| **NFR-03** | **Security** | การสื่อสารข้อมูลระหว่าง Client และ Server จะต้องผ่านโปรโตคอล HTTPS เพื่อความปลอดภัย |
| **NFR-04** | **Usability** | ผู้ใช้งานใหม่ควรเรียนรู้วิธีการใช้งานฟังก์ชันหลัก (เช่น การจองนัดหมาย) ได้ด้วยตนเองภายใน 5 นาที |
| **NFR-05** | **Reliability** | ระบบควรมีความเสถียรและสามารถใช้งานได้ตลอด 24 ชั่วโมง (Uptime 99%) |
| **NFR-06** | **Maintainability** | โค้ดที่พัฒนาต้องมีโครงสร้างที่ชัดเจน มีการเขียนคอมเมนต์ในส่วนที่ซับซ้อน เพื่อให้ง่ายต่อการแก้ไขและพัฒนาต่อในอนาคต |
